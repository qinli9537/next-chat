##  SSE 协议与流式解析

### SSE 是什么，和 HTTP 长连接以及 websocket 的区别
- **SSE (Server-Sent Events)** 是一种基于 HTTP 的单向实时通信技术，服务器可以主动向客户端推送事件流。使用 `text/event-stream` MIME 类型，客户端通过 `EventSource` API 接收数据。
- **HTTP 长连接** 是一种保持 TCP 连接复用的机制（通过 `Connection: keep-alive` 头部），减少频繁建立连接的开销，但仍然是传统的请求 - 响应模式，服务器不会主动推送数据。
- **WebSocket** 是一种全双工通信协议，在单个 TCP 连接上提供双向实时通信。客户端和服务器都可以主动发送消息，适合需要频繁交互的场景（如聊天室、在线游戏）。

### 三者的对比

| 特性 | SSE | HTTP 长连接 | WebSocket |
|------|-----|-------------|-----------|
| **通信方向** | 单向（服务器→客户端） | 双向（请求 - 响应） | 双向（全双工） |
| **协议** | HTTP/HTTPS | HTTP/HTTPS | ws:// 或 wss:// |
| **推送能力** | ✅ 服务器主动推送 | ❌ 需要轮询 | ✅ 双方主动推送 |
| **连接保持** | ✅ 自动重连 | ✅ 连接复用 | ✅ 长连接 |
| **数据格式** | 文本（支持自定义格式） | 任意 | 二进制/文本 |
| **浏览器支持** | 现代浏览器（不支持 IE） | 所有浏览器 | 所有现代浏览器 |
| **使用场景** | 通知、实时更新、流式响应 | 普通 Web 应用 | 聊天、游戏、协同编辑 |

### SSE 的优势

- ✅ **简单易用**：基于 HTTP，无需额外协议
- ✅ **自动重连**：内置断线重连机制
- ✅ **跨域支持**：支持 CORS
- ✅ **轻量级**：相比 WebSocket 更轻量
- ✅ **适合 AI 流式输出**：完美适配 LLM 的 token 流式生成

### SSE 的局限

- ❌ 仅支持单向通信（服务器→客户端）
- ❌ 不支持二进制数据
- ❌ 某些旧浏览器不支持（如 IE）

### AI产品选择SSE的原因

- 模型推理是**单向流式输出**，不需要双向通信，适合 SSE 协议，且基础设施成本最低。
